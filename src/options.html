How to configure

Simple configuration

The simplest configuration is for multiple target roles when you always intend to show the whole list. Target roles can be expressed with a role_arn or with both aws_account_id and role_name.

Optional parameters

  • label — A friendly display name shown in the popup instead of the profile name (the profile name stays searchable). Also used as the container name.
  • color — The RGB hex value (without the prefix '#') for the color of the header bottom border and around the current profile.
  • region — Changing the region whenever switching the role if this parameter is specified.
  • image — The uri of an image to use on top of any color attribute supplied. The color and image are not mutually exclusive.
[profile marketingadmin]
role_arn = arn:aws:iam::123456789012:role/marketingadmin
color = ffaaee

[anotheraccount]
aws_account_id = 987654321987
role_name = anotherrole
region=ap-northeast-1

[athirdaccount]
aws_account_id = 987654321988
role_name = athirdrole
image = "https://aesr.dev/img/150"

Complex configuration

More complex configurations involve multiple AWS accounts and/or organizations.

  • A profile specified by the source_profile of the others is defined as a base account.
  • If your account is aliased, you specify aws_account_alias in the base account.
  • If a role_name is specified in a base account it will also check for the role used to login to AWS. This selects a subset of accounts when using an SSO IdP. If a role name starts with AWSReservedSSO_, use only the permission set name.
  • A target role is associated with a base account by its source_profile naming the base account's profile.
  • Target roles can be a role_arn or both aws_account_id and role_name, and can pass the optional parameters.
  • If target_role_name is set in a base account, it is the default role name for each target role.
  • If target_region is set in a base account, it is the default region for each target role.
[organization1]
aws_account_id = 000011112222
aws_account_alias = your-account-alias

[Org1-Account1-Role1]
role_arn = arn:aws:iam::123456789012:role/Role1
source_profile = organization1

[Org1-Account1-Role2]
aws_account_id = 123456789012
role_name = Role2
source_profile = organization1

[Org1-Account2-Role1]
aws_account_id = 210987654321
role_name = Role1
source_profile = organization1

[baseaccount2]
aws_account_id = 000000000000

[Base2-Role1]
role_arn = arn:aws:iam::234567890123:role/Role1
source_profile = baseaccount2

;
; target_role_name example
;
[Org2-BaseAccount]
aws_account_id = 222200000000
target_role_name = Developer

[Org2-Account1-Developer]
aws_account_id = 222200001111
source_profile = Org2-BaseAccount

[Org2-Account2-Manager]
aws_account_id = 222200002222
role_name = Manager ; overrides target role name
source_profile = Org2-BaseAccount

If you sign-in a base account, target roles of the other base accounts are excluded.

The Show only matching roles setting is for sophisticated structures using AWS Organizations with Federated Logins. If you're logged in to the 'Developer' role in the master account, only member accounts with a role_arn ending in 'role/Developer' will be shown.

Firefox container tabs

On Firefox, each profile opens in its own color-coded container tab by default, so every AWS account keeps an isolated session — and switching from one role to another never has to go back through your base account. (Turn this off with the Open each profile in its own container tab setting.) Containers are created automatically, named after the profile (or its label), and colored from the profile color or, if none is set, a distinct color derived from the account. Tabs opened from a container tab stay in the same container.

To avoid a sign-in prompt, RoleDeck copies your current AWS login into the new container before switching, so it lands signed-in. Each account still keeps its own independent session afterwards. (This requires cookie access for AWS domains.)

A fresh container has no AWS session, so the first switch in a container asks you to sign in within that container; afterwards the sessions stay independent per container.

Optional parameters (per profile)

  • container — true opens this profile in a container named after the profile, false never uses a container, and any other value is used as a custom container name (lets profiles share one container). If omitted, the global Open each profile in its own container tab setting applies.
  • container_color — The container color: blue, turquoise, green, yellow, orange, red, pink, purple or toolbar. If omitted, the nearest container color is derived from the profile color.
  • container_icon — The container icon: fingerprint, briefcase (default), dollar, cart, circle, gift, vacation, food, fruit, pet, tree, chill or fence.
[profile production-admin]
role_arn = arn:aws:iam::123456789012:role/Admin
color = ff3333
container = true
container_color = red

[profile staging-admin]
role_arn = arn:aws:iam::210987654321:role/Admin
color = ffaa00
container = true
container_color = orange
container_icon = circle

[profile sandbox]
role_arn = arn:aws:iam::333344445555:role/Developer
container = shared-sandbox ; custom container name

Environment badges & production guardrails

RoleDeck tags each role with its environment and asks for confirmation before switching into production, so you don't land in the wrong account by accident.

  • env — production, staging or development. Shows a PROD / STG / DEV badge in the popup. If omitted, the environment is inferred from the profile name (names containing prod, staging, dev, etc.).
  • confirm — true always asks for confirmation before switching; false never does. If omitted, production profiles are guarded by default.
[prod-admin]
role_arn = arn:aws:iam::123456789012:role/Admin
env = production        ; PROD badge + confirm before switching

[billing-readonly]
aws_account_id = 999988887777
role_name = Billing
confirm = true          ; force a confirm even though it isn't prod

Favorites (the star icon) and recently used roles are managed right in the popup and need no configuration.

Deep links & shortcuts

  • start_url — switch straight to a specific console page. Set it to a full AWS console URL and selecting the profile lands you there instead of the page you're on.
[prod-monitoring]
role_arn = arn:aws:iam::123456789012:role/ReadOnly
env = production
start_url = https://us-east-1.console.aws.amazon.com/cloudwatch/home?region=us-east-1

In the popup: type to filter and press Enter to switch, ⭐ star favorites, and use Open all in the Favorites section to open each one in its own container tab. The Activity log and Managed config tools are in the panel on the left.

Settings reference

  • Hide account id hides the account_id for each profile.
  • Show only matching roles filters to only show profiles with roles that match your role in your master account.
  • Automatically renew a switched-role session before it expires re-assumes the role shortly before the 1-hour switch-role limit so you stay signed in, by silently reloading the tab to the page you're on. Because the tab reloads, unsaved input on the console page (e.g. a policy you're editing) can be lost. Renewal only works while your base login is still valid, and does not apply to multi-session (Prism) tabs.
  • Open each profile in its own container tab (Firefox only) opens every profile in a color-coded Firefox container named after the profile, without needing container = true on each profile. Profiles can opt out with container = false.
  • Automatic tab grouping for multi-Session (Chrome & Edge) organizes tabs from the same multi-session into tab groups named after the profile. Removing a group signs that session out.
  • Sign-in endpoint in current region uses a regional endpoint instead of signin.aws.amazon.com for slightly faster switching far from Virginia.
  • Automatically assume last assumed role re-assumes the last role on the next sign-in if you didn't return to the base account and sign out. (temporarily disabled)
  • Configuration storage — 'Sync' shares config between browsers via your account but can't store many profiles; 'Local' is the opposite.
  • Visual mode selects light or dark UI appearance.
